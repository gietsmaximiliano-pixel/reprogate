## Docker live-test status

Docker Desktop is installed and Docker Engine is running.

- `docker version`: passed; Docker Desktop `4.74.0`, client `29.4.3`, server engine `29.4.3`.
- Documented live command read from `docs/PUBLISH_GUIDE.md`:

```sh
node packages/cli/dist/index.js verify-safe-run examples/safe-run-smoke/reprogate.yml --yes --timeout 60 --cpus 1 --memory 512m
```

Live Docker commands executed:

- `node packages/cli/dist/index.js verify-safe-run examples/safe-run-smoke/reprogate.yml --yes --timeout 60 --cpus 1 --memory 512m`: passed. ReproGate printed the exact Docker command before execution, Docker ran `node smoke-check.js`, and the smoke output confirmed non-root UID/GID, no host secret env passthrough, read-only workspace, writable tmpfs, memory limit `536870912`, CPU limit `100000 100000`, disabled network, and successful completion.
- `"n" | node packages/cli/dist/index.js verify-safe-run examples/safe-run-smoke/reprogate.yml --timeout 60 --cpus 1 --memory 512m`: passed. ReproGate printed the exact Docker command, displayed the confirmation prompt, accepted `No`, and cancelled without running Docker.
- `node packages/cli/dist/index.js verify-safe-run examples/safe-run-smoke/reprogate-timeout.yml --yes --timeout 1 --cpus 1 --memory 512m`: passed as a negative timeout check. The command exited nonzero as expected, printed `ReproGate safe-run timed out after 1 seconds.`, and the harness measured `elapsedSeconds=1.86`, proving the timeout wrapper terminated the long-running command before it could finish.

Earlier attempted live checks found two issues and both were fixed:

- The previous documented Docker command targeted `examples/basic-node-bug/reprogate.yml`, whose `npm test` command fails by design to reproduce the bug. The documented live verification command now targets `examples/safe-run-smoke/reprogate.yml`.
- The original `timeout -s TERM <seconds> sh -lc <command>` wrapper did not terminate `node slow-check.js` reliably in `node:20-alpine`. Safe-run now uses a Node wrapper that parses commands into argv, spawns the child process group, and terminates that group on timeout.

## Known limitations

- Only schema version `0.1` is supported.
- Safe-run is intentionally conservative and rejects shell control operators rather than supporting complex scripts.
- AI adapters are mock-only and advisory; no real provider is implemented.
- Dashboard generation is static and local-only.
- `reprogate create` is interactive and was verified by typecheck/build, not by an automated terminal interaction in this audit.

### Node.js 20 action deprecation review

GitHub will force Node.js 24 for all action runtimes starting **2026-06-16**. The following was found and addressed:

| Action               | Old version | Change                                                               | Rationale                                                                                                                                                                       |
| -------------------- | ----------- | -------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `actions/checkout`   | `@v4`       | kept `@v4`; added `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24: true` job env | `checkout@v5/v6` changed credential storage; no backwards-compatible Node24 v4 patch exists                                                                                     |
| `pnpm/action-setup`  | `@v4`       | kept `@v4`; covered by `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24`          | Latest major is `v6.0.8`; upgrading requires careful testing of cache-path interactions with `setup-node`                                                                       |
| `actions/setup-node` | `@v4`       | upgraded to `@v5`                                                    | `v5.0.0` was the first release that runs on the Node.js 24 action runtime; `v6.0.0` broke `cache: pnpm` (limited automatic caching to npm only), so `@v5` is the correct target |

The `FORCE_JAVASCRIPT_ACTIONS_TO_NODE24: true` env var covers `checkout` and `pnpm/action-setup` immediately, matching the opt-in mechanism documented in the GitHub deprecation notice.
